# 背景

管理端及相关客户端登录能力目前以手机号为主，在需要区分「运维/管理账号」与「手机号用户」的场景下不够直观；同时存在为演示便利而预置的默认账号信息，不利于安全与交付形象。管理员初始化若仅依赖手工建库，不利于批量维护；若采用表格维护账号信息，又需要明确「表格可读可改」与「库内密码不可明文」之间的边界。

# 目标

1. 支持用户使用**手机号或账号**完成密码登录，且在交互上共用同一套标识输入与同一接口字段。
2. 支持用户继续使用**手机号**完成短信验证码登录（验证码路径不扩展为账号）。
3. 移除前端为演示而预置的默认手机号与默认密码，避免个人化或固定演示凭据残留。
4. 支持从 Excel 表 `vadmin_auth_user` 读取管理员初始化信息：表格中密码以**明文**维护以便运维，系统初始化入库时必须按安全方式加密存储。

# 要求

## 功能性要求

1. **密码登录**
   - 登录标识与密码在前端各仅一个输入框；与后端交互时标识与密码各对应**一个字段**（与现有「手机号+密码」接口形态保持一致或经设计说明的等价扩展）。
   - 后端根据标识内容判定类型：**若全为数字且符合国内手机号规则，则按手机号处理**；**账号须包含字母**（与手机号规则互斥时的判定细则以开发设计为准）。
   - **短信验证码登录**仅允许使用手机号作为登录标识。
2. **前端**：不得再默认填充演示用手机号（如历史环境中的 `15020221010`）及默认密码；各端（管理端、/uni 等）若存在同类预置，须一并按本需求清理或改为由环境/文档说明的本地调试方式（不写入默认提交代码路径）。
3. **管理员初始化**：从 Excel `vadmin_auth_user` 导入或等价初始化流程中，`password` 列允许以**明文**出现在表格中；执行入库初始化时，必须将密码转换为系统约定的**加密/哈希**形式存储，禁止以明文写入业务库用户表。

## 非功能性要求

1. 标识类型判定、格式校验失败、账号不存在或凭据错误等场景，对外错误提示策略须与现有安全与产品约定一致（避免过度泄露系统内部判断细节的行为在设计中明确）。
2. 国内手机号规则以当前产品约定为准；若后续支持国际号码，须单独需求与设计，不在本需求范围内默认扩展。
3. 若接口路径或字段名发生不兼容变更，须在开发设计与发布说明中单独列出迁移与兼容策略。
