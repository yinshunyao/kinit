# 背景

kinit-api 在运行过程中需要两类与「可观测与可运维」强相关的数据能力：一是对已认证接口的操作审计（供管理端查询谁在何时调用了哪些接口及结果）；二是系统管理中的定时任务配置、分组及与调度器联动的执行记录。当前实现中，上述数据主要落在 MongoDB；在部分部署或成本约束下，团队希望关闭 MongoDB，仅保留 MySQL 作为持久化存储。若关闭 Mongo 后上述能力不可用或行为不一致，将影响合规审计、问题排查与任务运维，因此需要明确「无 Mongo 模式下」的数据落库与功能等价要求。

# 目标

1. 支持在关闭 MongoDB 的配置下，操作审计日志仍能可靠写入并可在管理端按既有查询维度使用。
2. 支持在关闭 MongoDB 的配置下，定时任务相关的配置、分组、调度注册与执行记录等能力在 MySQL 上完整可用。
3. 明确关闭 Mongo 与仅使用 MySQL 时的配置约定及行为边界，避免运行期仍尝试连接 Mongo 或出现静默失败。

# 要求

## 功能性要求

1. 当 MongoDB 被配置为不启用时（与代码或配置中的开关语义一致，如 `MONGO_DB_ENABLE` 为假），原依赖 Mongo 的操作审计数据须改由 MySQL（或与主业务一致的关系型库）持久化；管理端「操作日志」相关能力须保持可用，查询与展示语义与开启 Mongo 时一致或经书面说明的差异可接受范围。
2. 同一配置下，原由 Mongo 承载的定时任务定义、分组、调度任务注册与执行记录等数据，须由 MySQL 表结构及数据访问层承接，后台任务管理与调度联动行为须完整可用。
3. 配置切换后，应用不得在无 Mongo 模式下仍初始化或强依赖 Mongo 连接；错误与降级行为须可诊断（如明确日志或启动失败原因），避免部分功能静默不可用。

## 非功能性要求

1. 对外 HTTP 接口路径与主要字段语义应尽量保持兼容；若有破坏性变更，须在开发设计与发布说明中单独列出。
2. 在预期数据量下，列表与常用筛选查询须具备可接受的响应；必要的数据库索引与分页策略须在开发设计中体现。
3. 若存在自 Mongo 迁移到 MySQL 的历史数据诉求，迁移策略、空库初始化与双写/切换窗口须在设计与运维文档中可追溯（本需求不规定具体迁移工具，但要求结论可执行）。
